# استقرار لوحة التحكم — سياسة تشغيل حرجة

هذا الملف مرجع تشغيلي رسمي لحماية لوحة التحكم من الأعطال أثناء العمل الفعلي.

## الهدف

- ضمان استمرارية الوصول إلى لوحة التحكم بدون انقطاع.
- تقليل مخاطر الأعطال الناتجة عن تغييرات مصادقة الدخول أو بيئة الإنتاج.
- توفير خطة استجابة فورية خلال 5 دقائق عند أي عطل.

## نطاق الحساسية

أي تغيير يخص العناصر التالية يعتبر "تغيير حرج":

- مسارات الأدمن (`/_hmap-int-9kz2/...` أو ما يعادلها)
- مكونات المصادقة: `AdminLogin`, `AdminAuthHashGate`, `adminAuth`
- إعدادات Supabase Auth أو RLS
- متغيرات Vercel:
  - `VITE_SUPABASE_URL`
  - `VITE_SUPABASE_ANON_KEY`
  - `VITE_ADMIN_EMAIL`
  - `VITE_ADMIN_PORTAL_BASE` (إن استُخدم)

## قواعد ثابتة (لا تُكسر)

1. لا تعديل على بيئة الإنتاج أثناء ذروة التشغيل.
2. أي تغيير حرج يمر أولاً على بيئة اختبار/Preview.
3. لا نشر لتغييرات حرجة بدون اختبار يدوي لمسار الأدمن كاملًا.
4. لا حذف حسابات Bootstrap Admin بدون بديل مؤكد.
5. لا استخدام مفاتيح `service_role` في الواجهة الأمامية.

## حسابات الطوارئ

يجب الإبقاء على حسابي وصول طوارئ (Bootstrap) على الأقل:

- `admin@halaqmap.com`
- `ritchstar4@gmail.com`

ويجب أن يكون كلاهما:

- موجودًا في Supabase Auth
- مؤكد البريد (Confirmed)
- بكلمات مرور قوية ومعلومة لصاحب الصلاحية

## فحص ما قبل أي تغيير حرج (Checklist)

نفّذ جميع البنود قبل النشر:

- [ ] الدخول إلى رابط الأدمن يعمل: `/#/_hmap-int-9kz2/in`
- [ ] الانتقال إلى لوحة التحكم يعمل: `/#/_hmap-int-9kz2/ctrl`
- [ ] لا أخطاء Console حرجة أثناء فتح لوحة التحكم
- [ ] build محلي ناجح (`npm run build`)
- [ ] migration المطلوبة منفذة في Supabase
- [ ] متغيرات Vercel صحيحة ولم تتغير بالخطأ
- [ ] يوجد Commit مستقر واضح للرجوع السريع

## فحص ما بعد النشر (Smoke Test)

بعد كل نشر، خلال 2-3 دقائق:

1. افتح الرابط:
   - `https://www.halaqmap.com/#/_hmap-int-9kz2/in`
2. سجّل دخولًا فعليًا بحساب أدمن.
3. افتح:
   - نظرة عامة
   - الطلبات
   - المدفوعات
   - غرفة القيادة
   - الإعدادات
4. تأكد عدم ظهور شاشة بيضاء أو إعادة توجيه خاطئة.

## خطة الطوارئ (5 دقائق)

إذا تعطلت لوحة التحكم في الإنتاج:

1. إيقاف أي تغييرات إضافية فورًا.
2. تنفيذ Rollback لآخر Deploy مستقر في Vercel.
3. التحقق من متغيرات البيئة (خصوصًا مفاتيح Supabase).
4. التحقق من رابط الأدمن مع `#` وصحة `portal base`.
5. الدخول بحساب Bootstrap Admin للتأكد من أن العطل ليس صلاحيات فقط.
6. فتح Incident داخلي وتوثيق السبب الجذري خلال نفس اليوم.

## سياسة التغيير

- تغييرات `auth/admin` تُدمج في أوقات صيانة محددة.
- كل تعديل حساس يجب أن يحتوي:
  - سبب التغيير
  - خطة اختبار
  - خطة رجوع
- ممنوع تعديلات "سريعة" مباشرة على الإنتاج بدون توثيق.
